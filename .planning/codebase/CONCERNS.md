## Tech Debt

**Broken and superseded lead scraper:**
- Issue: `outputs/scrape_leads.py` uses the Scrapeless API (`api.scrapeless.com/api/v1/scraper/request`), which is a different provider than the current Apify-backed `new-leads-scraper` skill. The script targets wrong-niche queries (`"content marketing agency USA"`, `"social media marketing agency"`, `"video content agency"`) — not personal-injury attorneys. It is also broken by design: the Scrapeless async path polls a task ID but the response key `local_results` does not match Scrapeless's actual response schema.
- Files: `outputs/scrape_leads.py`
- Impact: Running this script produces zero usable PI leads and may waste Scrapeless credits. It is a dead artifact.
- Fix approach: Delete `outputs/scrape_leads.py`. All scraping is now handled through the `new-leads-scraper` skill (`~/.agents/skills/`), which uses Apify MCP tool calls, not a standalone Python script. Do not attempt to fix the old script.

**Wrong-ICP lead list (general NYC law, not PI):**
- Issue: `uploads/law_firm_leads.csv` contains 30 rows scraped from NYC Google Maps. Category breakdown: 22 "Law firm" (generic), 3 "Attorney" (generic), 2 "Personal injury attorney", 1 "Legal services", 1 "Real estate attorney", 1 "Patent attorney". Only 2 of 30 rows match the target ICP (PI firms in FL/TX/CA/NY/GA, 2–15 attorneys). The file was generated by a prior scraping session aimed at a different outreach niche.
- Files: `uploads/law_firm_leads.csv`
- Impact: Any outreach or enrichment built on this list targets the wrong audience. Sending PI-focused cold emails to real-estate or patent attorneys is a deliverability and reputation risk.
- Fix approach: Do not use `uploads/law_firm_leads.csv` as a source list. Run the `new-leads-scraper` skill with query `"personal injury attorney"` scoped to one metro (Tampa, Houston, Miami, or Atlanta) to generate a fresh, correct list. Archive or delete the existing CSV.

**Email generator uses pre-PI copy:**
- Issue: `outputs/lawfirm_emails.js` contains the lead list from `uploads/law_firm_leads.csv` (Weitz & Luxenberg, Lebedin Kofman, etc. — general NYC firms) hardcoded in a `const leads = [...]` array. The email body copy pitches "AI systems that automate client intake for small law firms" with a generic "quick question about your intake process" subject. There is no mention of personal injury pain points: speed-to-lead, missed calls after accidents, Google Ads lead leakage, or the sub-60-second qualify flow.
- Files: `outputs/lawfirm_emails.js`, `outputs/lawfirm_outreach_emails.docx`
- Impact: These assets are misaligned with the PI funnel strategy. Using them as-is sends the wrong message to the wrong firms.
- Fix approach: Once a correct PI lead list exists, rebuild the `leads` array with PI firms. Rewrite the email body to reference: speed-to-lead after an auto accident inquiry, qualifying callers for case merit, and a live demo CTA ("text this number, watch it qualify you in 5 seconds"). The `.docx` is a compiled artifact — update the `.js` source and regenerate.

**Hardcoded ephemeral session paths in all JS generators:**
- Issue: Every output generator file uses absolute paths tied to a long-dead Docker/sandbox session: `require('/sessions/hopeful-trusting-ramanujan/mnt/outputs/node_modules/docx')` and `fs.writeFileSync('/sessions/hopeful-trusting-ramanujan/mnt/outputs/...')`. That session path no longer exists.
- Files: `outputs/lawfirm_pamphlet.js`, `outputs/lawfirm_emails.js`, `outputs/medspa_pamphlet.js`, `outputs/realestate_pamphlet.js`
- Impact: All four `.js` generator scripts fail with `MODULE_NOT_FOUND` when run from the current environment. The `.docx` files were produced during the original session and are already committed artifacts; the generators cannot reproduce them without path fixes.
- Fix approach: Replace the hardcoded `require(...)` path with a relative `require('./node_modules/docx')` (since `outputs/node_modules/` exists locally). Replace `fs.writeFileSync` paths with relative output paths. The `node_modules` in `outputs/node_modules/` is already present and usable.

**Scope pollution — non-lawfirm artifacts in the lawfirm directory:**
- Issue: `outputs/` contains `medspa_pamphlet.js`, `medspa_automation_pamphlet.docx`, `realestate_pamphlet.js`, `realestate_automation_pamphlet.docx`, and `ai_workflow_leads.md` (content/video marketing agency leads, dated May 13 2026). These are artifacts from earlier pitches for different niches.

## Security Considerations

**Plaintext API key committed in `scrape_leads.py`:**
- Risk: `outputs/scrape_leads.py` line 4 contains `SCRAPELESS_API_KEY = "sk_3lz..."` — a live credential hardcoded directly in a committed source file. Any person or system with read access to this repository or directory can use this key against the Scrapeless API.
- Files: `outputs/scrape_leads.py`
- Current mitigation: None. There is no `.gitignore` in this repository. The key is fully exposed.
- Recommendations: (1) Immediately rotate the Scrapeless API key at the Scrapeless dashboard. (2) Remove the hardcoded key from `scrape_leads.py` and replace with `os.environ.get("SCRAPELESS_API_KEY")`. (3) Add a `.gitignore` that excludes `.env`, `*.env`, and any file matching `*secret*` or `*credential*`. (4) Add a `.env` file for local secrets and add it to `.gitignore` before the next commit.

**No `.gitignore` — sensitive files are unprotected from accidental commit:**
- Risk: The repository has no `.gitignore`. This means `audit.jsonl` (1.49 MB, 320-line session transcript), `.audit-key` (binary, 75 bytes — appears to be a cryptographic key/HMAC secret), `.claude/.credentials.json` (OAuth tokens for 30+ third-party integrations: Slack, HubSpot, GitHub, Linear, ZoomInfo, Outreach, Apollo, Salesforce-adjacent tools, BigQuery, etc.), and `.claude/.claude.json` (contains `userID`, `oauthAccount`, and session caches) are all unprotected.
- Files: `.audit-key`, `audit.jsonl`, `.claude/.credentials.json`, `.claude/.claude.json`
- Current mitigation: None.
- Recommendations: Create `.gitignore` immediately with at minimum: `.env`, `.env.*`, `audit.jsonl`, `.audit-key`, `.claude/`, `outputs/node_modules/`, `uploads/*.csv`.

**`audit.jsonl` contains full session transcripts with potential PII and tool call inputs:**
- Risk: `audit.jsonl` (320 lines, 1.49 MB) is a complete log of every AI conversation and tool call in this project. It contains: full user message content, tool call inputs to Apify actors (including scraping configuration), rate-limit events with account metadata, session IDs, and client platform info. The file also contains 94 lines matching email/phone/contact fields — likely from scraped lead data that passed through tool responses logged here. It includes `_audit_hmac` fields suggesting it is HMAC-verified by `.audit-key`. Neither file is gitignored.
- Files: `audit.jsonl`, `.audit-key`
- Current mitigation: Files are local-only as long as no `git add` is run.
- Recommendations: Add both to `.gitignore` immediately. Consider whether `audit.jsonl` needs to be retained at all; if yes, store outside the repo directory.

**`.claude/.credentials.json` contains OAuth tokens for 30+ services:**
- Risk: This file holds active OAuth tokens for integrations across Slack, HubSpot, GitHub, Linear, ZoomInfo, Outreach, Apollo, Atlassian, Figma, Box, Intercom, Datadog, Monday.com, ClickUp, Klaviyo, Ahrefs, Supermetrics, Amplitude, Canva, and others. If this file is committed to git or exposed, all connected accounts are compromised.
- Files: `.claude/.credentials.json`
- Current mitigation: Local only, no `.gitignore`.
- Recommendations: Add `.claude/` to `.gitignore`. Consider whether these credentials belong in the project directory at all vs. the global Claude config location.

## Compliance Risks

**No AI guardrail implementation exists yet — critical gap:**
- Risk: SYSTEM_PLAN.md section 5 states a non-negotiable compliance requirement: "The AI qualifies and schedules only. It must never give legal advice, assess case merit, or make representations for the firm." This guardrail is required in every demo and production system prompt. As of this analysis, **no demo system prompt file exists anywhere in the repository**. There is no Twilio webhook, no n8n workflow, no FastAPI app, and no system prompt document enforcing the qualify-only boundary.
- Files: None exist yet. The gap is total for Subsystem 3 (demo).
- Impact: If a demo is built quickly without an explicit, tested system prompt, the AI could respond to a prospective client's case question in a way that constitutes unauthorized legal advice — exposing the law firm client to bar complaints and the builder to liability for enabling it.
- Recommendations: Before any demo goes live, create a `prompts/qualify-only-system-prompt.md` file (or similar) with explicit restrictions. Test against adversarial inputs: "Do I have a good case?", "Should I sue?", "How much is my case worth?". The system prompt must deflect all of these and route to booking only. Include a disclaimer line in every AI reply (e.g., "I'm an intake assistant — I don't give legal advice or evaluate your case.").

**Email outreach does not yet mention demo CTA — compliance of implied promise:**
- Risk: The current outreach email copy (`outputs/lawfirm_emails.js`) offers a "30-Day Free Pilot" in the footer but does not include the demo CTA ("text this number, watch it qualify you in 5 seconds"). When recipients ask to see the product, there is no live demo to show. Promising a working system that does not exist creates a credibility and possibly misrepresentation risk.
- Files: `outputs/lawfirm_emails.js`, `outputs/lawfirm_outreach_emails.docx`
- Current mitigation: None — the demo has not been built yet.
- Recommendations: Do not send outreach until the demo number is live and tested. Per SYSTEM_PLAN.md build order, the demo (Phase 1) must be completed before outreach (Phase 2).

## Fragile Areas

**`outputs/` directory has no package.json script or runner — JS generators are manually executable only:**
- Files: `outputs/lawfirm_pamphlet.js`, `outputs/lawfirm_emails.js`, `outputs/medspa_pamphlet.js`, `outputs/realestate_pamphlet.js`, `outputs/package.json`
- Why fragile: `outputs/package.json` has no `scripts` block — only a `dependencies` entry for `docx`. There is no `npm run build` or equivalent. The generators are run directly with `node outputs/lawfirm_pamphlet.js` but currently fail due to the broken absolute `require()` path. Once path is fixed, the `.docx` output will be regenerated to a path that may differ from where the committed `.docx` files currently live.
- Safe modification: Fix the `require()` and `fs.writeFileSync()` paths to use `path.join(__dirname, ...)` relative paths before running. Always run from the `outputs/` directory.

**n8n workflow library is generic, not PI-configured:**
- Files: `workflows/` (190+ generic templates across 190 subdirectories)
- Why fragile: The `workflows/` directory contains only generic n8n templates imported from a shared library — not a single PI-specific workflow. The Twilio workflows present (`workflows/Twilio/`) are generic SMS triggers unrelated to PI intake. There is no Twilio → Claude → Calendar booking workflow anywhere. The Calendly workflows (`workflows/Calendly/`) connect to Notion, Mautic, or Noop — not to a Claude qualifier.
- Safe modification: Do not modify existing generic workflow templates. When building the demo, create new workflow files under a `workflows/PI-Demo/` directory to keep them separate from the template library.

## Dependencies at Risk

**Scrapeless API key is committed and exposed:**
- Risk: `SCRAPELESS_API_KEY` in `outputs/scrape_leads.py` is a live credential. Anyone who reads the file can make authenticated requests to Scrapeless at the account owner's expense.
- Impact: Financial (unexpected API charges) and security (potential data harvesting using the compromised key).
- Migration plan: Rotate key immediately. Move to environment variable. Script is otherwise abandoned in favor of Apify.

**`outputs/node_modules/` is committed (likely) but has no lockfile discipline:**
- Risk: `outputs/node_modules/` is present in the repo with `docx@^9.x` and its dependencies. There is no `.npmrc` and no engine pinning in `outputs/package.json`. The `package-lock.json` exists but is not enforced by any CI.
- Impact: A future `npm install` without the lockfile could upgrade `docx` to a breaking version, silently changing generated `.docx` formatting.
- Migration plan: Add `outputs/node_modules/` to `.gitignore` and rely on `npm ci` to install from `package-lock.json` consistently.
